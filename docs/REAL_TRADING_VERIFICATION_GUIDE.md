## 二、准备工作

### 2.1 环境配置

确保 `.env` 文件包含以下配置：

```bash
# 币安 API 配置
BINANCE_API_KEY=your_api_key_here
BINANCE_API_SECRET=your_api_secret_here

# 交易配置
INITIAL_CAPITAL=10000
MAX_POSITION_SIZE=0.3
MAX_SINGLE_POSITION=0.2
PAPER_TRADING=true
COMMISSION_RATE=0.001
DEFAULT_SYMBOL=BTCUSDT
DEFAULT_INTERVAL=1h

# 数据库配置
DB_HOST=localhost
DB_PORT=5432
DB_NAME=binance
DB_USER=postgres
DB_PASSWORD=your_password
```

### 2.2 依赖安装

```bash
pip install python-binance
pip install python-dotenv
```

## 三、测试网验证步骤

### 3.1 创建测试网 API 密钥

1. 访问 [币安测试网](https://testnet.binance.vision/)
2. 注册并登录账号
3. 创建 API 密钥（启用交易权限）
4. 保存 API Key 和 Secret

### 3.2 配置测试网

修改 `.env` 文件：

```bash
USE_TESTNET=true
BINANCE_API_KEY=testnet_api_key
BINANCE_API_SECRET=testnet_api_secret
```

### 3.3 运行测试网验证脚本

```bash
python testnet_verification.py
```

## 四、实盘验证注意事项

### 4.1 风险控制

- **初始资金**：建议使用总资金的 5-10%
- **单笔仓位**：不超过总资金的 5%
- **止损设置**：每笔交易设置 2-3% 止损
- **日亏损限制**：设置 5% 的日亏损上限
- **总仓位控制**：总仓位不超过 30%

### 4.2 安全措施

1. **API 权限限制**：
   - 仅启用交易权限
   - 禁用提现权限
   - 设置 IP 白名单

2. **监控和日志**：
   - 启用详细日志
   - 实时监控账户状态
   - 定期检查交易记录

3. **紧急停止**：
   - 准备手动紧急停止机制
   - 设置自动熔断机制
   - 准备备用资金方案

## 五、实盘验证流程

### 5.1 第一天：观察期

- 目标：熟悉系统运行
- 操作：只观察不交易
- 任务：
  - 监控市场数据获取
  - 验证趋势分析准确性
  - 观察策略匹配结果
  - 记录系统响应时间

### 5.2 第二天：最小资金测试

- 目标：验证下单和执行
- 操作：最小单位交易
- 任务：
  - 测试市价单下单
  - 测试限价单下单
  - 验证订单执行
  - 确认盈亏计算

### 5.3 第三天：杠杆交易测试

- 目标：验证杠杆功能
- 操作：低杠杆（2x）交易
- 任务：
  - 测试做多功能
  - 测试做空功能
  - 验证强平价格计算
  - 确认保证金管理

### 5.4 第四天：策略交易

- 目标：验证策略执行
- 操作：自动化策略交易
- 任务：
  - 验证趋势识别
  - 确认策略匹配
  - 检查风险控制
  - 记录策略表现

### 5.5 第五天：全功能测试

- 目标：验证完整交易流程
- 操作：完整交易周期
- 任务：
  - 端到端流程测试
  - 性能压力测试
  - 异常情况处理
  - 系统稳定性验证

## 六、紧急情况处理

### 6.1 紧急停止条件

触发紧急停止的情况：
- 单笔亏损超过 5%
- 日亏损超过 10%
- 系统异常无法恢复
- API 连接持续失败
- 市场剧烈波动超出预期

### 6.2 紧急停止步骤

1. **立即停止程序**：
   ```bash
   # Ctrl+C 中断程序
   # 或使用紧急停止脚本
   python emergency_stop.py
   ```

2. **检查账户状态**：
   - 查询当前持仓
   - 检查未成交订单
   - 确认账户余额

3. **手动处理持仓**：
   - 必要时手动平仓
   - 取消所有未成交订单
   - 记录当前状态

4. **分析问题原因**：
   - 检查交易日志
   - 分析错误原因
   - 修复问题缺陷

5. **决定是否继续**：
   - 问题修复后重新评估
   - 必要时延长测试周期
   - 确保持续安全运行
