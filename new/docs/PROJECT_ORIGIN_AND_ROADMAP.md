# 项目创意起源与演进路线

> **Execute = Alpha, Prediction ≠ Alpha**
> 执行就是Alpha，预测不等于Alpha

---

## 一、项目起源：核心创意

### 问题背景

绝大多数量化交易系统都专注于**价格预测**：
- 预测下一步价格上涨还是下跌
- 用机器学习/深度学习拟合历史价格
- 信号生成后直接下单

但**高频做市**和**执行优化**领域，真相是：

> **Alpha 不在预测，而在与队列和时间的博弈**

你的信号对不对只占一半，另一半是：
- 你能不能在正确的位置成交？
- 你能不能避开毒流量？
- 你能不能拿到好的成交价格？
- 你能不能在队列中生存到成交？

### 核心洞察：Level 2.5 影子撮合

传统方案的问题：

| 方案 | 问题 |
|------|------|
| 历史回放（静态）| RL 会利用未来信息，学到 ghost alpha |
| 全局市场冲击 | 算力爆炸，小型团队无法承受 |
| 简单撮合 | 不真实，过拟合严重 |

**我们的方案：Level 2.5 Shadow Matching**

```
你的订单是透明的（Shadow）
历史市场数据照常播放
你的订单不会改变历史订单簿
但系统会计算你的"影子订单"在历史轨迹中能否成交
```

**优势**：
- ✅ 独立开发者/小型团队可负担
- ✅ 比静态回放更真实
- ✅ 比全局冲击更可计算
- ✅ 训练环境 ≈ 实盘执行环境

## 二、核心设计思想演进

### 阶段 1：确定性队列衰减（v1）

```python
# 初始思路：确定性衰减
Q_{t+1} = max(0, Q_t - V_trade - V_cancel_ahead)

# Q <= 0 → 成交
```

**问题**：RL 会利用确定性，卡精准位置，学到不现实的策略。

### 阶段 2：随机衰减 + 概率建模（v2）

引入随机扰动防 exploit：

```python
Q_{t+1} = Q_t - V_trade - P(cancel_ahead) * V_cancel + noise
P(cancel_ahead) = queue_position / total_volume
```

**关键改进**：
- 不再确定性知道撤单位置
- 引入随机噪声打破确定性
- 防止 RL 记住历史路径

### 阶段 3：Hazard Rate 概率撮合（v3）→ **当前基线**

从"是否成交"升级到"成交概率分布"：

```
λ = base_rate × exp(-α × queue_ratio) × (1 + β × OFI) × (1 + γ × trade_intensity)
P(fill in Δt) = 1 - exp(-λ × Δt)
```

**质变**：
- ✅ 连续时间概率模型
- ✅ 队列位置、订单流、交易强度都影响成交概率
- ✅ 更接近真实市场的随机性

### 为什么异构？

- **Go**：天生适合低延迟网络编程，gc停顿可控，内存占用可控
- **Python**：生态丰富，PyTorch/NumPy/Pandas 方便研究和迭代
- **mmap**：零拷贝，避免序列化开销，微秒级通信延迟

## 五、设计原则：防骗防exploit

### 三条铁律

1. **破坏确定性**：任何时候都加随机扰动，不让 RL 记住历史路径
   - 队列位置扰动
   - 时间戳扰动
   - 成交概率扰动

2. **逆向选择成本**：毒流量必须提前预测并惩罚，不能只事后算账
   ```
   R = ... - λ2 × P(toxic) × fill_size
   ```

3. **概率而非确定**：不说"一定会成交"，说"成交概率是多少"
   - Hazard Rate 建模
   - 部分成交支持
   - 生存分析框架

### 防 exploit 检查清单

- [ ] 队列衰减引入随机噪声 ✅
- [ ] 撤单位置概率建模 ✅
- [ ] Hazard Rate 概率成交 ✅
- [ ] 时间戳扰动 ✅
- [ ] 毒流预测前置到状态 ✅
- [ ] 延迟进入状态空间 ✅

## 七、实盘递进指南：安全第一

### 步骤 1：影子跟单（Dry Run）

```
Go 引擎连接实盘
Python 生成动作
但不下真实单
只记录：如果下单会是什么成交价格
运行 1-2 周，观察 fill_quality 和 adverse_selection
```

**检查点**：
- `fill_quality` 均值 < 0 → 合格
- `adverse_selection` 均值 ≈ 0 → 合格
- `latency` 95% < 200ms → 合格

### 步骤 2：小资金试跑

- 总资金 ≤ 5%
- 单笔 ≤ 1%
- 只跑一个交易对（推荐 BTCUSDT）
- 低频交易（每天几笔）

### 步骤 3：逐步扩容

合格后逐步增加：
- 资金 → 10% → 20% → 50%
- 交易对 → +ETH → +SOL → ...
- 频率 → 逐步提高

### 安全红线（绝对不能破）

```go
if drawdown > 0.02 {  // 回撤 2%
    kill_switch()    // 立即停止
    os.Exit(1)
}
```

## 八、核心认知：一句话总结

> 你不是在开发"预测价格的模型"
> 你是在构建"一个会学习如何在队列中生存并盈利的系统"

**Alpha 来源**：
- 不是猜对方向
- 而是比别人拿到更好的成交位置
- 避开毒流量
- 吃返佣
- 在正确的时机入场

## 十、清理：已淘汰的设计思路

以下思路已被淘汰，不需要再考虑：

1. **纯静态历史回放** → ❌ 淘汰（RL 会 exploit 确定性）
2. **全局市场冲击建模** → ❌ 淘汰（算力爆炸，小型团队无法承担）
3. **单一标量动作空间** → ❌ 淘汰（升级到三维：方向+激进度+大小）
4. **确定性成交** → ❌ 淘汰（升级到概率 Hazard Rate）
5. **事后毒流惩罚** → ❌ 淘汰（升级到前置预测）
6. **Latency 只在环境，不进状态** → ❌ 淘汰（Latency 必须进状态）

## 十一、关键判断：决定成败的分水岭

> 当你跑完 Phase 1-2，你已经拥有了**交易系统 + 可观测性**。
> 下一步，**谁更快发现自己在亏钱，谁就能活下去**。

下一个决定胜负的阶段：**在线学习 + 自动降级 + 自愈**。

系统应该能：
1. 发现自己执行质量变差 → 自动降级
2. 发现新模型更好 → A/B 测试验证 → 自动切换
3. 发现毒流 → 自动收缩仓位
4. 发现回撤触发 → 自动停止

这就是**自我演化的交易系统**。
