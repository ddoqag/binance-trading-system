## ADR-001: 使用 mmap 共享内存进行 Go-Python 通信

**状态**: ✅ 已接受

**日期**: 2026-03-15

**背景**: 需要实现Go执行引擎与Python AI层之间的低延迟通信。

**决策**: 使用mmap共享内存配合序列锁（Sequence Lock）实现零拷贝通信。

**理由**:
- 延迟: mmap < 1μs vs gRPC > 1ms
- 零拷贝: 避免数据序列化/反序列化开销
- 无锁: 序列锁实现无锁同步

**替代方案**:
- gRPC: 延迟过高 (~10ms)
- ZeroMQ: 额外依赖，复杂度增加
- Unix Domain Socket: 需要拷贝

**影响**:
- 需要精心设计内存布局（缓存行对齐）
- 增加跨平台兼容性复杂度
- 需要严格的版本控制

## ADR-002: SAC算法选择

**状态**: ✅ 已接受

**日期**: 2026-03-10

**背景**: 需要选择适合高频交易场景的强化学习算法。

**决策**: 使用Soft Actor-Critic (SAC)作为基础算法。

**理由**:
- 连续动作空间支持
- 样本效率高（适合高成本采样）
- 自动温度调节，平衡探索与利用
- Twin Critics减少过估计

**替代方案**:
- PPO: 实现简单但样本效率较低
- DQN: 仅支持离散动作
- TD3: 类似SAC但无熵正则化

**影响**:
- 需要仔细调整温度参数
- 需要较大的回放缓冲区
- 训练稳定性较好

## ADR-003: 三层异构架构

**状态**: ✅ 已接受

**日期**: 2026-03-20

**背景**: 需要在性能和灵活性之间取得平衡。

**决策**: 采用 Go (执行层) + Python (决策层) + mmap (通信) 的三层架构。

**理由**:
- Go: 微秒级性能，适合高频执行
- Python: 丰富的ML生态，适合AI研发
- 分离关注点，独立演进

**替代方案**:
- 纯Go: ML生态不足
- 纯Python: 性能不足
- Rust: 学习曲线陡峭

**影响**:
- 增加系统复杂度
- 需要维护两种语言代码库
- 调试难度增加

## ADR-004: 混合风控架构

**状态**: ✅ 已接受

**日期**: 2026-03-25

**背景**: 需要平衡RL智能体的灵活性和系统的安全性。

**决策**: 采用"RL决策 + 规则风控覆盖"的混合架构。

**理由**:
- RL负责捕捉市场机会（进攻）
- 规则风控负责硬性约束（防守）
- 独立风控系统拥有最终裁决权

**替代方案**:
- 纯RL风控: 安全性不足
- 纯规则风控: 灵活性不足

**影响**:
- 增加系统延迟（风控检查）
- 需要维护两套系统
- 提高整体安全性

## ADR-005: Phase 1优先执行层

**状态**: ✅ 已接受

**日期**: 2026-03-30

**背景**: 需要确定项目升级的第一步重点。

**决策**: Phase 1优先强化执行层（实盘接入、状态机、WAL）。

**理由**:
- 实盘交易是后续优化的基础
- 没有实盘数据，RL训练效果有限
- 风险控制必须优先于收益优化

**替代方案**:
- 优先优化RL算法: 无实盘意义不大
- 优先完善监控: 重要性较低

**影响**:
- 延迟AI功能的完善
- 确保系统基础稳固
- 降低实盘风险

## ADR-006: 暂不支持杠杆交易（Phase 1）

**状态**: ✅ 已接受

**日期**: 2026-03-30

**背景**: 项目范围需要明确。

**决策**: Phase 1专注于现货交易，杠杆交易延后到Phase 3。

**理由**:
- 降低初期复杂度
- 风险控制更可控
- 现货系统可作为杠杆基础

**替代方案**:
- 同时开发: 风险过高
- 优先杠杆: 复杂度过高

**影响**:
- 限制初期收益空间
- 降低爆仓风险
- 延长项目周期
